Meta E4 Interview Loop Structure

The Meta E4 iOS loop typically consists of 4–5 rounds conducted over one day (virtual or onsite). Knowing what each round tests lets you prepare with precision.

Round 1 — Coding (45 min)
Data Structures & Algorithms
2 problems in 45 minutes. You choose the language (use Python). Problems are typically Medium difficulty, occasionally one Easy + one Hard. You must solve both completely, handle edge cases, and state time/space complexity.
Problem Solving Coding Speed Communication Bug-Free Code
Round 2 — Coding (45 min)
Data Structures & Algorithms (second interviewer)
Same format, different interviewer and problems. Both coding rounds carry equal weight. If you nail one and bomb the other, it's typically a no-hire. Consistency matters.
Same signals as Round 1
Round 3 — Behavioral (45 min)
Meta Values & Leadership
5–7 behavioral questions mapped to Meta's core values. Expect: "Tell me about a time you moved fast", "Describe a technical disagreement", "Give an example of measurable impact." STAR format is mandatory. Every answer needs a quantified result.
Meta Values Alignment Self-Awareness Impact & Ownership Collaboration
Round 4 — System Design (45 min)
Mobile System Design (may be product-specific)
Design a mobile feature end-to-end: "Design Instagram's feed", "Design an image caching library", "Design a chat client." Expected to cover: requirements, high-level architecture, data model, networking, caching, offline, and performance. At E4 the bar is lower than E5 — show you can think through trade-offs, not that you've built it before.
Architecture Thinking Trade-off Analysis iOS Knowledge Scalability Awareness
Round 5 — iOS Deep Dive (sometimes combined with design)
iOS Platform Knowledge
Not always a separate round — sometimes woven into coding or design. Topics: Swift concurrency, memory management, UIKit/SwiftUI trade-offs, performance profiling, threading. They want to see that you understand the platform, not just that you can code on it.
Platform Depth Debugging Approach Production Experience

Hiring committee note: After the loop, the packet goes to a hiring committee. They weigh all rounds roughly equally. A "strong hire" in coding can't compensate for a "no hire" in behavioral. You need to clear the bar in every round.

Union Find (Disjoint Set)
class UnionFind: def __init__(self, n): self.parent = list(range(n)) self.rank = [0] * n def find(self, x): if self.parent[x] != x: self.parent[x] = self.find(self.parent[x]) # path compression return self.parent[x] def union(self, x, y): px, py = self.find(x), self.find(y) if px == py: return False if self.rank[px] < self.rank[py]: px, py = py, px self.parent[py] = px if self.rank[px] == self.rank[py]: self.rank[px] += 1 return True
Trie (Prefix Tree)
class TrieNode: def __init__(self): self.children = {} self.is_end = False class Trie: def __init__(self): self.root = TrieNode() def insert(self, word): node = self.root for c in word: if c not in node.children: node.children[c] = TrieNode() node = node.children[c] node.is_end = True def search(self, word): node = self.root for c in word: if c not in node.children: return False node = node.children[c] return node.is_end

Swift DSA Cheat Sheet

Swift-specific implementations for common DSA structures. Useful for iOS depth questions where they want to see Swift idioms.

Min Heap (Swift has no built-in)
struct MinHeap<T: Comparable> { private var items: [T] = [] var count: Int { items.count } var isEmpty: Bool { items.isEmpty } var peek: T? { items.first } mutating func insert(_ val: T) { items.append(val) siftUp(items.count - 1) } mutating func remove() -> T? { guard !isEmpty else { return nil } items.swapAt(0, items.count - 1) let val = items.removeLast() if !isEmpty { siftDown(0) } return val } private mutating func siftUp(_ i: Int) { var child = i var parent = (child - 1) / 2 while child > 0 && items[child] < items[parent] { items.swapAt(child, parent) child = parent; parent = (child - 1) / 2 } } private mutating func siftDown(_ i: Int) { var parent = i while true { let l = 2 * parent + 1, r = 2 * parent + 2 var smallest = parent if l < items.count && items[l] < items[smallest] { smallest = l } if r < items.count && items[r] < items[smallest] { smallest = r } if smallest == parent { break } items.swapAt(parent, smallest); parent = smallest } } }
Union Find
class UnionFind { private var parent: [Int] private var rank: [Int] init(_ n: Int) { parent = Array(0..<n) rank = Array(repeating: 0, count: n) } func find(_ x: Int) -> Int { if parent[x] != x { parent[x] = find(parent[x]) } return parent[x] } func union(_ x: Int, _ y: Int) -> Bool { var px = find(x), py = find(y) if px == py { return false } if rank[px] < rank[py] { swap(&px, &py) } parent[py] = px if rank[px] == rank[py] { rank[px] += 1 } return true } }
Trie
class TrieNode { var children: [Character: TrieNode] = [:] var isEnd = false } class Trie { let root = TrieNode() func insert(_ word: String) { var node = root for ch in word { if node.children[ch] == nil { node.children[ch] = TrieNode() } node = node.children[ch]! } node.isEnd = true } func search(_ word: String) -> Bool { var node = root for ch in word { guard let next = node.children[ch] else { return false } node = next } return node.isEnd } }
LRU Cache
class LRUCache { private class Node { let key: Int; var val: Int var prev: Node?; var next: Node? init(_ k: Int, _ v: Int) { key = k; val = v } } private var map: [Int: Node] = [:] private let capacity: Int private let head = Node(0, 0), tail = Node(0, 0) init(_ capacity: Int) { self.capacity = capacity head.next = tail; tail.prev = head } func get(_ key: Int) -> Int { guard let node = map[key] else { return -1 } remove(node); insertAtHead(node) return node.val } func put(_ key: Int, _ value: Int) { if let node = map[key] { remove(node); node.val = value; insertAtHead(node) } else { let node = Node(key, value); map[key] = node; insertAtHead(node) if map.count > capacity { let lru = tail.prev!; remove(lru); map[lru.key] = nil } } } private func remove(_ n: Node) { n.prev?.next = n.next; n.next?.prev = n.prev } private func insertAtHead(_ n: Node) { n.next = head.next; n.prev = head; head.next?.prev = n; head.next = n } }

Mock Interview Self-Evaluation Rubric

After every practice session, score yourself on these signals. Meta interviewers use a similar framework. Be honest — the point is to find weaknesses before the real interview.

1
Problem Solving
Did I identify the right approach within 5 minutes? Did I consider multiple approaches and pick the optimal one? Did I break the problem into sub-problems?
2
Communication
Did I verbalize my thought process throughout? Did I explain WHY I chose my approach, not just WHAT I did? Did I ask clarifying questions before jumping in?
3
Code Quality
Is my code clean and readable? Did I use meaningful variable names? Could I code without relying on autocomplete? Did I avoid unnecessary complexity?
4
Complexity Analysis
Did I state time and space complexity immediately after proposing a solution — without being asked? Did I know WHY it's that complexity, not just the answer?
5
Edge Cases & Testing
Did I proactively identify edge cases (empty input, single element, duplicates, overflow)? Did I trace through my code with a test case? Did I catch bugs before running?
6
Speed
Did I finish both problems within 45 minutes? Did I avoid spending too long on one approach before pivoting? Could I code the solution without hesitation once I had the approach?

Scoring yourself: For each signal, give yourself Strong (did it naturally), Mixed (did it partially or with prompting), or Weak (missed it). If you have more than one "Weak" after a mock, focus your next study session on that specific signal.

Product-Sense Scenarios

Meta values product-minded engineers. Practice talking through these scenarios — they can appear in system design rounds or as follow-up questions in coding rounds.

Feed
"Design Instagram's feed to work offline. What happens when the user opens the app with no connectivity?"
Key points to hit: Local cache of feed data (SQLite or Core Data), cached images on disk, show stale content with "last updated" indicator, queue interactions (likes/comments) for replay on reconnect, handle cache invalidation when connectivity returns, pagination cursor persistence.
Messaging
"How would you ensure message ordering in a WhatsApp-like chat when the user has intermittent connectivity?"
Key points to hit: Local sequence numbers, server-assigned timestamps, optimistic display with pending state, conflict resolution on sync, vector clocks or Lamport timestamps for ordering, retry queue with exponential backoff, idempotency keys to prevent duplicates.
Media
"Instagram Reels needs to preload videos for smooth scrolling. How do you balance preloading with memory and bandwidth constraints?"
Key points to hit: Preload next 2-3 videos only, cancel preloads when scroll direction changes, use low-quality preview initially then upgrade, monitor memory pressure via os_proc_available_memory, pause preloading on cellular if user preference, disk cache with size cap for recently viewed.
Infrastructure
"Your app's crash rate spiked 3x after a release. Walk me through how you'd investigate and respond."
Key points to hit: Check crash reports (MetricKit, Firebase Crashlytics), identify top crasher by stack trace, check if it correlates with specific device/OS/feature, consider rollback vs hotfix decision, use feature flags to disable the problematic feature, post-mortem process, add monitoring to catch earlier next time.
Feed
"Facebook's feed has 10+ different cell types (text, photo, video, link preview, ad, story tray, etc.). How do you architect the collection view layer for 60fps?"
Key points to hit: DiffableDataSource with section snapshots, CompositionalLayout for heterogeneous sections, register cell classes per type (avoid giant switch), pre-calculate cell heights in background, image downsampling via ImageIO, prefetch API for upcoming cells, self-sizing cells with cached intrinsic sizes, avoid Auto Layout in hot paths.
Messaging
"How would you implement the 'typing...' indicator in Messenger? Consider performance and battery impact."
Key points to hit: WebSocket for real-time events, debounce typing events (don't send every keystroke), auto-expire indicator after 5s timeout, batch typing events for group chats, only subscribe to typing events for visible conversations, pause subscription when app is backgrounded, minimize payload size.
Media
"Design the image upload flow for an Instagram post. The user selects 10 photos — how do you handle upload reliability?"
Key points to hit: Background URLSession for upload continuity, chunked/resumable uploads, compress and resize before upload, upload queue with retry logic, progress tracking per image, handle partial failures (some uploaded, some failed), persist upload state across app restarts, show optimistic post with upload progress.

Common E4 Mistakes

These are the patterns that specifically sink E4 candidates. Awareness alone prevents most of them.

🔇
Coding in silence
The #1 reason for "Mixed" or "No Hire" feedback at E4. You solve the problem but don't explain your thinking. The interviewer can't give you credit for thought process they can't see.
Fix: Narrate like a cooking show — "I'm reaching for a hash map here because I need O(1) lookup for duplicates..."
🏃
Jumping straight into code
Starting to type before understanding the problem or discussing your approach. Leads to wrong solutions that eat up time.
Fix: Spend 3-5 minutes on approach. Say: "Before I code, let me walk through my approach..." Confirm with the interviewer before writing.
🔨
Over-engineering the solution
Building a class hierarchy when a function would do. Adding error handling for cases that can't happen. Writing helper functions for one-time operations.
Fix: Solve it simply first. If the interviewer wants more, they'll ask. The simplest correct solution is always the best starting point.
🧪
Not testing your code
Saying "I think this works" without tracing through a test case. Meta interviewers expect you to verify your solution — this is a scored signal.
Fix: After coding, trace through with a concrete example. Say: "Let me walk through this with input [2,7,11,15], target 9..."
😰
Panicking when stuck
Going silent for 3+ minutes, erasing everything and starting over, or saying "I don't know" without attempting to reason through it.
Fix: Verbalize your struggle: "I'm stuck on the transition between states. Let me think about what happens when..." Interviewers often give hints to candidates who communicate well.
📅
Skipping behavioral prep
Spending all prep time on DSA and cramming STAR stories the night before. Behavioral is weighted equally — a weak behavioral round can override strong coding.
Fix: Write your 5 STAR stories by Week 4. Practice them aloud weekly. Record yourself and listen back — you'll catch vague answers.
❓
Not asking clarifying questions
Assuming constraints that weren't stated. Not asking about input size, edge cases, or whether the input is sorted. Meta interviewers deliberately leave problems under-specified to test this.
Fix: Always ask: "What's the expected input size?", "Can there be duplicates?", "Is the array sorted?", "Should I optimize for time or space?"
🎯
Not stating complexity unprompted
At E3, the interviewer might ask about complexity. At E4, you're expected to state it yourself. Not doing so signals a gap in self-sufficiency.
Fix: After describing your approach, immediately say: "This is O(n) time and O(n) space because..." Make it automatic.

E4 Offer & Negotiation Context

Understanding the compensation structure helps you evaluate and negotiate effectively. These are approximate ranges for E4 (IC4) based on publicly available data (levels.fyi, Blind). Ranges vary by location.

Component Range (US, 2024-2025) Notes
Base Salary $150K – $185K Less negotiable — stays within band
Equity (RSU) $80K – $200K/yr Vests over 4 years, most negotiable component
Sign-on Bonus $20K – $50K One-time or split over 2 years
Annual Bonus 10-15% of base Performance-dependent
Total Comp $250K – $380K Median ~$300K for E4
01
Competing offers are the strongest lever. Even an offer from a smaller company gives you data. Meta recruiters will ask directly — "Do you have other offers?"
02
Negotiate equity, not base. Base salary has a narrow band. RSU grants have much more room. Ask: "Is there flexibility on the equity component?"
03
Don't share your current salary. In many states it's illegal for them to ask. Focus on your target total comp, not what you currently make.
04
The level matters more than the initial offer. Coming in at E4 vs E5 has a larger long-term impact than any negotiation on the initial package. If you think you might be E5, discuss it before the loop.
05
Timeline pressure works both ways. If Meta gives you an exploding offer (e.g., "decide by Friday"), you can ask for a reasonable extension. A week or two is normal.

Interview Day Logistics

What to expect on the day itself, whether virtual or onsite.

💻
Virtual setup: Meta uses CoderPad (or similar) for coding rounds. You code in a shared editor — no autocomplete, no running code (usually). Test with the link beforehand. Use a wired connection if possible. Have a glass of water nearby.
🏢
Onsite: You'll code on a whiteboard or laptop depending on the office. Arrive 15 minutes early. You'll have a host who walks you between rooms. Lunch is usually with an engineer (not scored, but be personable).
⏰
Schedule: Expect 4-5 back-to-back 45-minute rounds with 15-minute breaks between some. The full loop takes about 4-5 hours. It's a marathon — eat a good breakfast and bring snacks.
📝
What to have ready: Pen and paper (or tablet) for scratching out ideas during virtual rounds. Your 5 STAR stories outlined on a notecard for quick reference during behavioral. A question to ask each interviewer ("What's the team structure?", "What's a recent technical challenge?").
🫠
When you're stuck: Don't go silent. Say: "I'm thinking through the edge cases here..." or "I have a brute force approach — let me describe it and then optimize." Interviewers at Meta are trained to give hints to candidates who communicate well. Asking for a hint is better than stalling in silence.
🧘
The day before: Don't cram. Review your STAR stories once, glance at your cheat sheet, then stop. Get 8 hours of sleep. A rested brain outperforms a crammed one. You've already put in the work — trust your preparation.
📨
After the interview: Send a short thank-you email to your recruiter (not the interviewers). Ask about timeline for the decision. Typical: feedback within 1 week, hiring committee within 2 weeks, offer within 3 weeks. It's OK to follow up if you haven't heard back after 2 weeks.

How to Use This Prep Plan

A structured approach to landing your Meta E4 iOS offer in 12 weeks

Overview

This tool covers two pillars of the Meta E4 iOS interview: DSA coding (12 weeks of LeetCode-style problems) and iOS engineering depth (18 topic cards covering everything from ARC to push notifications).

The plan is divided into 4 phases that build on each other. Don't skip ahead — each phase assumes mastery of the previous one.

Phase 1
Foundations
Weeks 1–4 · Arrays, Hashing, Linked Lists, Stacks. These cover ~60% of Meta DSA questions.
Phase 2
Core Algorithms
Weeks 5–8 · Binary Search, Trees, Graphs, Heaps. The medium-difficulty backbone of Meta interviews.
Phase 3
Advanced Topics
Weeks 9–11 · DP, Backtracking, Tries & Design. Harder problems that separate E4 from E3.
Phase 4
Interview Readiness
Week 12 · Mock interviews, timed practice, final review of Meta favorites.

Recommended Daily Routine

Morning (1–1.5 hrs)
Solve 2 new DSA problems from the current week. Use the built-in timer set to 25 minutes per problem. If you can't solve it in 25 min, study the solution, then re-solve from scratch the next day.
Afternoon (30–45 min)
Study 1 iOS engineering topic from the cards. Read the linked resources, then close them and try to explain the concept aloud. Use the notes feature to write key points in your own words.
Evening (20–30 min)
Review completed problems flagged by the spaced repetition system. Re-solve them without looking at your previous solution. Check the review schedule at the top of the DSA tab for what's due.

Meta E4 Interview Strategy

01
Communicate constantly. Meta interviewers score you on communication as much as correctness. Verbalize your thought process: "I'm considering a hash map here because lookup is O(1)..." Don't code in silence.
02
State complexity immediately. After proposing a solution, say the time and space complexity before they ask. At E4, this is expected without prompting.
03
Handle edge cases proactively. Before coding, list edge cases: empty input, single element, duplicates, negative numbers, overflow. At E4, interviewers expect you to catch these yourself.
04
Know the Meta favorites. Problems tagged with "★ Meta" in this plan are confirmed or frequently reported Meta questions. Prioritize these. If you're short on time, solve only the Meta-tagged problems.
05
Bridge DSA to iOS. When discussing solutions, connect them to real product scenarios: "This LRU cache is essentially how NSCache works" or "This graph traversal is how we'd model a social network." Meta values product-minded engineers.
06
Prepare STAR stories early. Don't leave behavioral prep for the last week. Write 4–5 STAR stories covering: technical leadership, conflict resolution, a project failure, measurable impact, and cross-team collaboration. Practice them weekly.
07
Simulate real conditions. During Week 12, do full mock sessions: 2 problems in 45 minutes, on a whiteboard or plain text editor (no autocomplete). Use the 45-minute timer preset.
08
Don't over-prepare. Solving 300 LeetCode problems is less effective than deeply understanding 100. For each problem, know the pattern, the optimal solution, and at least one alternative approach. Quality over quantity.

Weeks 1–4 Complete all Easy/Medium problems. Build speed. Target: solve Easy in 10 min, Medium in 20 min.
